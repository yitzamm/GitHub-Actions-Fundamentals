Git CI/CD Pipeline with Docker

REPO SET-UP

Clone the repo locally:

git clone https://github.com/nanuchi/my-project.git

Change the remote origin:

git remote -v
git remote remove origin
git remote add origin https://github.com/yitzamm/GitHub-Actions-Fundamentals
git remove -v

Push to your repo using WSL:

git branch -m master main (rename the main branch)
git branch
sudo apt update
sudo apt install gh
sudo apt install ubuntu-wsl xdg-utils
sudo apt install wslu
gh auth login
gh auth setup-git
gh auth status
git push -u origin main

NOTE: We're using gh auth to authenticate through the browser. This can also be done via SSH. Powershell has a Git Credential Manager (GCM) pre-installed that uses a pop-up window for the Git credentials to be entered.

Build fail due to an application/dependency config issue. New Gradle requires implementation group rather than compile group.

compile      -> implementation
testCompile  -> testImplementation
runtime      -> runtimeOnly

BUILD AND PUSH DOCKER IMAGE


ERROR: failed to build: failed to solve: openjdk:8-jre-alpine: failed to resolve source metadata for docker.io/library/openjdk:8-jre-alpine: docker.io/library/openjdk:8-jre-alpine: not found

FIX: Since JDK, Gradle and Spring Boot were all updated to newer versions, the image used in the Dockerfile was flagged as deprecated during the build. Changed the FROM openjdk:8-jre-alpine line to FROM eclipse-temurin:17-jre-alpine.

NOTE: Your runtime container must match your build environment.

ERROR: Logging into Docker registry docker.io... /bin/sh: docker: command not found Error: Command failed: docker login -u *** --password-stdin docker.io /bin/sh: docker: command not found

Fix: Isolate the Docker push to ubuntu only while keeping the cross-platform builds. Docker images are Linux-based by default so the workflow failed when trying to build and push the Docker image on the Mac operating system. Linux containers need a Linux kernel whereas Windows/MacOS emulate Linux (extra layer). For that reason, Docker image builds are typically executed on Linux runners because containers rely on the host kernel, and Linux provides native, consistent behavior that matches most production environments.
